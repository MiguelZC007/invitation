# GGA Rules — Invitation Project

## Scope
- Review changes for a Next.js + TypeScript project using Atomic Design, i18n, and strict testing quality gates.

## Mandatory gates
1. Never allow commits targeting `main` or `develop` directly.
2. Enforce conventional commits only (`feat|fix|refactor|test|docs|chore`).
3. Require `pnpm` usage (reject npm/yarn instructions).
4. Require passing quality checks before merge:
   - `pnpm lint`
   - `pnpm test:run`
   - `pnpm coverage` (minimum 80%)

## TypeScript and code quality
- `strict` mode expectations: no `any`, no unsafe assertions without justification.
- Prefer `type` over `interface` per project convention.
- No barrel exports (`index.ts`) for component directories.

## Frontend architecture (strict)
- Enforce hexagonal boundaries when applicable:
  - `domain` (pure business logic)
  - `application` (use-cases)
  - `adapters` (infrastructure)
  - `ui/components/templates` (presentation)
  - `app` (routing/composition)
- Reject business logic in `page.tsx`, templates, and presentational components.
- Require ports/adapters for new infrastructure dependencies.

## UI/UX constraints
- Keep Atomic Design structure (`atoms`, `molecules`, `organisms`).
- Preserve responsive behavior (mobile-first Tailwind classes).
- Enforce accessibility: semantic HTML, keyboard support, and proper ARIA labels.

## i18n
- No hardcoded user-facing strings in localized screens/templates.
- Prefer translation keys through the existing `next-intl` flow (`messages/es.json`, `messages/en.json`).

## Testing expectations by change type
- Domain/application changes: unit tests for business behavior.
- Adapter changes: integration/contract-oriented tests.
- UI/template changes: interaction + accessibility tests.

## Review output format
- Classify findings as: `critical`, `warning`, `suggestion`.
- For each finding include: file, rationale, and concrete fix direction.
- End with: `PASS` or `NEEDS_CHANGES`.
